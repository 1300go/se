## 1. 서론

### 1.1 문서 목적 및 범위
본 문서의 목적은 소프트웨어 공학 과목의 프로젝트 기반 테스트 보장 체계를 수립하기 위해, 클라우드 기반 파일 관리 시스템 **'Mini Drive'**의 테스팅을 디자인하고 그 결과를 기술하는 데 있다. 본 문서에서는 프로젝트의 요구사항 정의서 및 분석서를 바탕으로 테스트 범위, 구체적인 테스트 케이스, 차후 결함 발생 시의 예외 처리 기준을 명시한다.

### 1.2 프로젝트 개요
Mini Drive는 조직 내 팀원 간의 파일 버전 혼선과 보안 유출 문제를 해결하기 위해 200명 규모의 특정 협업 환경에 최적화된 프라이빗 클라우드 파일 공유 시스템이다. 최대 2GB 이상의 대용량 파일 전송, 계층형 디렉터리 권한 제어, OTP/이메일 다중 인증, 메타데이터 기반 고속 검색, 그리고 최소 10개 이상의 버전 관리 및 시점 복구 기능을 핵심으로 제공한다.

### 1.3 용어 정의
| 용어 | 설명 |
| :---: | :--- |
| **MFA (다중 인증)** | 아이디와 비밀번호 외에 OTP나 이메일 인증 등을 추가하여 보안 단계를 강화하는 방식. |
| **감사 로그 (Audit Log)** | 시스템 내에서 누가, 언제, 어떤 파일에 접근하여 수정하거나 삭제했는지 기록한 보안 증적 자료. |
| **메타데이터 (Metadata)** | 파일의 이름, 크기, 날짜, 확장자 등 데이터를 설명하는 핵심 정보이며 고속 검색의 효율성을 높임. |

## 2. 테스트 개요

### 2.1 테스트 범위
테스트는 시스템 내부 구조를 모른다는 가정하에, 사용자 시점에서 요구사항 만족 여부를 검증하는 **블랙박스 테스트(Black-box Test)**로 이루어진다. 테스트의 구체적인 범위는 아래와 같다.

| 범위 | 내용 |
| :--- | :--- |
| **Web UI 클라이언트** | 브라우저/서버사이드 동작 정확성 확인 및 별도 매뉴얼 없이 5분 이내 숙달 가능한 직관적 UI 검증 |
| **Backend API** | RESTful API 엔드포인트 동작 제어, 2GB 이상 대용량 파일 스트리밍 전송의 신뢰성 확인 |
| **Database 레이어** | PostgreSQL(메타데이터, 이력 관리) 및 Redis(캐싱, OTP 세션) 데이터 입출력 및 연동 시험 |

### 2.2 테스트 항목 및 통과 기준

#### 2.2.1 기능별 테스트 항목
| 기능별 테스트 항목 | 통과 기준 |
| :--- | :--- |
| **회원가입/로그인 기능** | 올바른 계정 정보 양식만 가입에 성공하며, 로그인 시 입력값 검증이 올바르게 수행되어야 함. |
| **OTP 다중 인증 보안** | 발급된 6자리의 일회용 인증 번호가 일치해야만 메인 화면 접근 권한이 부여됨. |
| **파일 업로드/다운로드** | 최대 2GB 파일 업로드가 가능하며, 네트워크 단절 후 재연결 시 이어받기가 안정적으로 성공해야 함. |
| **폴더 및 권한 관리** | 계층형 폴더 이동/삭제가 가능하고, 지정된 역할(읽기/편집)에 따라 접근 제어가 엄격히 동작해야 함. |
| **스마트 검색 기능** | 파일명, 날짜 등의 메타데이터 조건 검색 시 200명 규모 환경에서도 1초 이내에 결과가 반환되어야 함. |
| **버전 관리 및 복구** | 파일 수정 시 신규 버전 스냅샷이 자동 생성되며, 최소 10개 버전 중 선택한 시점으로 복구가 완벽히 수행되어야 함. |
| **감사 로그 기록** | 파일 접근, 수정, 삭제 등의 비즈니스 행위 발생 시 실시간으로 감사 DB에 누락 없이 적재되어야 함. |

#### 2.2.2 비기능별 테스트 항목
| 비기능별 테스트 항목 | 통과 기준 |
| :--- | :--- |
| **브라우저 호환성 테스트** | 별도 프로그램 설치 없이 Chrome, Safari, Edge 최신 브라우저 환경에서 시스템이 동일하고 정상적으로 작동함. |
| **대용량 성능/부하 테스트** | 약 200명의 사용자가 동시 접속하여 검색 및 파일 다운로드를 요청해도 속도 저하 없이 안정적인 서비스를 보장함. |
| **보안 테스트** | 로그인 및 다중 인증을 거치지 않은 비인가 사용자는 파일 데이터 및 시스템 기능에 접근할 수 없음. |

## 4. 테스트 예외사항

### 4.1 중단 기준과 재개 조건
* **중단 기준:** 테스트 수행 과정 중 파일 전송 모듈 오류로 인해 가용 호스트 커널 세션이 다운되거나, 백엔드 API 및 PostgreSQL/Redis 데이터베이스 커넥션이 완전히 유실되어 트랜잭션 무결성이 깨지는 치명적인 결함이 발견될 경우 테스팅을 즉각 중단한다.
* **재개 조건:** 인프라 컨테이너 환경 재빌드 및 데이터베이스 스키마 초기화가 완료되고, 테스트용 샘플 데이터셋(200인 기준 도메인 계정 및 폴더 트리)이 에러 없이 마운트되었음이 검증되면 중단되었던 테스트 케이스 단계부터 정상 재개한다.

## 5. 테스트 환경

### 5.1 테스트 환경 인프라 구성
본 시스템의 테스트는 안정적인 데이터 흐름과 보안 무결성을 검증하기 위해 클라이언트, API 서버, 영속성/캐싱 DB 레이어로 명확히 분리하여 통합 테스트 환경을 구축한다.

| 레이어 분류 | 구성 도구 및 기술 스택 | 테스트 목적 및 주요 검증 내역 |
| :--- | :--- | :--- |
| **프론트엔드 (Client)** | Chrome / Safari / Edge 브라우저 | 멀티 브라우저 호환성 및 파일 드래그 앤 드롭 UI 인터페이스 정상 동작 여부 검증 |
| **백엔드 (API Server)** | Flask 또는 FastAPI 프레임워크 | HTTP/TLS 기반 RESTful API 통신 상태 및 2GB+ 대용량 파일 스트리밍 중단 시의 제어 로직 검증 |
| **메타데이터 (Main DB)** | PostgreSQL 데이터베이스 Engine | 파일 고유 정보, 계층 구조 디렉터리 맵, 실시간 파일 접근/수정에 대한 보안 감사 로그 적재 무결성 검증 |
| **캐싱 및 보안 (Sub DB)** | Redis 인 메모리 데이터 스토리지 | 로그인 세션 정보 유지 관리 및 이중 보안 토큰(OTP 발급 번호 일치 확인용) 임시 캐싱 성능 검증 |

## 1. 서론

### 1.1 문서 목적 및 범위
본 문서의 목적은 소프트웨어 공학 과목의 프로젝트 기반 테스트 보장 체계를 수립하기 위해, 클라우드 기반 파일 관리 시스템 **'Mini Drive'**의 테스팅을 디자인하고 그 결과를 기술하는 데 있다. 본 문서에서는 프로젝트의 요구사항 정의서 및 분석서를 바탕으로 테스트 범위, 구체적인 테스트 케이스, 차후 결함 발생 시의 예외 처리 기준을 명시한다.

### 1.2 프로젝트 개요
Mini Drive는 조직 내 팀원 간의 파일 버전 혼선과 보안 유출 문제를 해결하기 위해 200명 규모의 특정 협업 환경에 최적화된 프라이빗 클라우드 파일 공유 시스템이다. 최대 2GB 이상의 대용량 파일 전송, 계층형 디렉터리 권한 제어, OTP/이메일 다중 인증, 메타데이터 기반 고속 검색, 그리고 최소 10개 이상의 버전 관리 및 시점 복구 기능을 핵심으로 제공한다.

### 1.3 용어 정의
| 용어 | 설명 |

## 4. 테스트 예외사항

### 4.1 중단 기준과 재개 조건
* **중단 기준:** 테스트 수행 과정 중 파일 전송 모듈 오류로 인해 가용 호스트 커널 세션이 다운되거나, 백엔드 API 및 PostgreSQL/Redis 데이터베이스 커넥션이 완전히 유실되어 트랜잭션 무결성이 깨지는 치명적인 결함이 발견될 경우 테스팅을 즉각 중단한다.
* **재개 조건:** 인프라 컨테이너 환경 재빌드 및 데이터베이스 스키마 초기화가 완료되고, 테스트용 샘플 데이터셋(200인 기준 도메인 계정 및 폴더 트리)이 에러 없이 마운트되었음이 검증되면 중단되었던 테스트 케이스 단계부터 정상 재개한다.
